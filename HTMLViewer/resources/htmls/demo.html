Zetes Wings

An easy way to make
an independent GUI app

Long long time ago when every PC on a desk in a schoolboy's room was a holy grail, creating a piece of independent code was quite a simple task. Actually, in DOS almost all the code was completely independent of... well... everything. Programs were linked statically, there were no “system APIs”, only some programmable BIOS interrupts — for mouse, hard disks and so on... Of course there was the dos.h file... A ridiculously tiny thing for such a global name (compare to windows.h).

Of course there were many disadvantages in that. Every app was written from scratch. You didn't have any help in such simple tasks as drawing on screen, reading mouse or keyboard input or making sounds. You even had to support every non-standard device in your app... Every app used to become a small OS.

Then in 1995-2000 modern OSes appeared. In most cases it was Windows. But there also was Mac OS which later switched into x86 processors and turned into OS X. And even linux became GUI-friendly with its Xorg and GTK+. The new era has arised — the era of desktop GUI applications.

Unfortunately the cross-platform paradigm was still too heavy for real use. Everyone who tried to make an application for multiple OSes at the same time used to implement their own visual components toolkit or use a default one... Most system APIs was very different and many wrapping code should have been written. In addition Windows wasn't a POSIX-complying OS, so many API functions (even the low-level ones like file I/O) were implemented differently in it and in other desktop OSes.

In 1995 Sun Microsystems released Java. The language had a very beautiful slogan: “write once, run anywhere”. And it was partially true — Java API was made as a thick wrapping to the OS API — it unified the features and let almost every application work on every OS. It was even suitable as a web scripting language, but failed cause of some technical issues. But, in addition, Java had a huge disadvantage. Its default GUI toolkit — Swing — was ugly. It was slow and looked like an alien on every OS. Java applications had a visible mark on them: “written in Java”. That feeling was intensified by the fact that you had to download and install Sun JRE in order to run any Java app — no matter, was it a huge server or a small game.

* * *

The author of Zetes started with a dream. He dreamed of a framework that would let a programmer to create a GUI application that would target multiple platforms simultaneously, has a beautiful UI on each of them and doesn't make the user to download any additional proprietary components. Modern computers allow you to easily download 50 or even 100 megabytes of code. That size should hold everything necessary for an application to work well (except the system API libraries which are present on every installation). That's why Zetes was made.

Of course, most parts of the framework were not originated by its author. But thankfully every one of them have either BSD or Apache licenses that allows everyone to use the code in their projects without any doubt.
